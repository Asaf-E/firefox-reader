Project goal

Build a Firefox extension that:

reads novel/chapter text from the normal webpage,
detects the main reading content,
sends paragraphs or sections to an LLM for readability improvement,
renders the improved result in the extension’s own clean reading view,
preserves meaning, names, terms, and tone as much as possible.

Use standard Firefox WebExtensions architecture: content scripts to read page content, background logic for coordination/API calls, and an extension page for the clean reader UI. Firefox supports temporary installation during development from about:debugging, and regular extensions on iPhone/iPad are not supported, while Android does support extensions with constraints.

Product behavior
User flow
User opens a novel chapter page.
User clicks the extension button.
Extension extracts the main reading content from the current page.
Extension opens its own clean reader page.
Reader page shows:
title if available,
source URL,
extracted original paragraphs,
button to improve selected paragraph / visible section / whole chapter.
Improved text is shown beside or instead of original.
User can switch between:
original,
improved,
side-by-side.
User can choose rewrite mode:
light cleanup,
natural prose,
strict meaning-preserving.
Results are cached per URL + paragraph hash.
High-level architecture
Components
1. manifest.json

Defines extension metadata, permissions, action button, content scripts, background script/service worker, storage, and extension pages. Firefox supports content scripts and extension APIs through the WebExtensions model.

2. Content script

Runs in the page context and:

inspects DOM,
identifies likely main reading container,
extracts visible text blocks,
sends extracted data to background or opens reader page with serialized data.

Content scripts can read and modify web page content when host permissions are granted.

3. Background script

Responsible for:

handling toolbar click,
coordinating message passing,
opening the extension reader page,
making or proxying API requests,
managing cache and settings.

Background scripts are the event-handling core of an extension.

4. Reader page

An extension-owned page like reader.html that:

renders extracted text in a clean reading layout,
provides controls,
triggers rewrite requests,
displays original and improved content.

Extension APIs are available in extension pages, not just background/content scripts.

5. Storage layer

Use browser storage for:

user settings,
glossary,
cache,
rewrite mode,
model/provider settings.

Suggested technical decisions
1. Use your own reader page, not page replacement

Do not replace the original site DOM with improved text as your primary UX.
Safer approach:

extract content from source page
show result in extension-owned reader page

Why:

less breakage from site scripts
easier layout control
easier caching and settings
easier security model
2. Personal-use security model vs product security model

For personal use:

local API key in extension settings is acceptable if you understand the risk

For serious distribution:

use your own backend
do not expose provider keys in the client
3. Start with manual trigger only

Do not auto-run on every page load at first.

4. Prefer paragraph-level control first

Whole-chapter rewriting should come later.

5. Build glossary support early

This matters more than fancy UI.

Prompt constraints

no summarization
no censorship unless required by safety
no added exposition
output plain text only
